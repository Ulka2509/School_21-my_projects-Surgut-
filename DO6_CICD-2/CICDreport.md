### Part 2. Сборка

>Написала этап для **CI** по сборке приложений из проекта *C3_SimpleBashUtils*:
Создала и описала файл .gitlab-ci.yml
Файл .gitlab-ci.yml - это файл конфигурации для настройки и определения задач и пайплайнов непрерывной интеграции
и непрерывной доставки (CI/CD) в GitLab. В этом файле описывается, какие шаги должны выполняться автоматически при
каждом пуше кода в репозиторий. После добавления .gitlab-ci.yml файла в репозиторий, GitLab CI/CD будет автоматически
создавать пайплайны и выполнять задачи согласно настройкам. Появляется возможность видеть результаты выполнения задач
в веб-интерфейсе GitLab, а также настраивать уведомления, автоматическое развертывание и другие аспекты CI/CD процесса.
> В файле _gitlab-ci.yml_ добавила этап запуска сборки через мейк файл из проекта с сохранением в произвольную директорию
 со сроком 30 дней.

Для того, что бы установить соединение мужду двумя виртуальными машинами, необходимо внести изменения в конфигурационный
файл netplan, адаптеры машин настроить через внутреннею сеть. 
Для того чтобы внести изменения в netplan файл и сохранить их необходимо создать файл 99-disable-network-config.cfg в дириктории /etc/cloud/cloud.cfg.d с содержанием `network: {config: disabled}`.

- ![basic_ci_cd](CICD/219.png)
- ![basic_ci_cd](CICD/221.png)
- ![basic_ci_cd](CICD/220.png)

 Пропинговала.
- ![basic_ci_cd](CICD/222.png)
На первой ВМ командой `docker run -it --rm ubuntu:latest`
вошла в интерактивный режим. `apt-get update && apt-get install -y openssh-client` установила службу ssh внутри образа контейнера.
Написала bash-скрипт, который при помощи ssh и scp копирует файлы, полученные после сборки (артефакты), в директорию /usr/local/bin второй виртуальной машины.
- ![basic_ci_cd](CICD/224.png)
- В файле gitlab-ci.yml добавить этап запуска написанного скрипта
- ![basic_ci_cd](CICD/225.png)
Для того, что бы артефакты перенеслись на вторую ВМ, необходимо под пользователем gitlab runner (`sudo su gitlab-runner`) скопировать, сгенерированный ssh ключ (`ssh-keygen`).
Скопировала файл с ключем на вторую ВМ `scp /home/gitlab-runner/.ssh/id-ed25519.pub rosa@192.168.0.8:/home/rosa/.ssh`.
На второй машине вошла в директорию и переименовала файл ключа `mv id-ed25519.pub authorized_keys`.
- ![basic_ci_cd](CICD/231.png)
Теперь с первой машины на вторую можно войти через `ssh rosa@192.168.0.8` без ввода пароля.

Пользоватоль gitlab-runner, под которым происходит копирование файлов на вторую ВМ не имеет прав root. В директорию, указанную по заданию, usr/local/bin копировать файлы может только пользователь с правами root.
Пользоватоль gitlab-runner не имеет доступ к это директории. Создала пользователя deploy, изменила права доступа для директории `sudo chown -R deploy /usr/local/bin`
Для того, что бы директория была доступна для копирования и записи ввела команду `sudo cmod a+w /usr/local/bin`.
